# Flutter Structure Standards

这是 `flutter-code-standards` 的 companion skill。只在入口 skill 路由到这里，或用户明确要求这一类结构规范时使用。

## 核心原则

- 先理解项目现有架构，再按当前改动范围做局部设计，不把每次改动都放大成全局重构。
- 默认追求够用、直接、可维护，不把“抽象更多”误当成“设计更好”。
- 维护已有代码时，默认遵循当前文件和项目既有风格；只有用户明确要求时，才按本规范重构既有实现。

## 结构健康检查

- 如果当前页面、弹窗、ViewModel 或辅助类出现职责混乱、层级不清、重复实现、方法过长或命名含糊，及时纠正。
- 如果套用规范本身导致文件拆得过碎、职责反而分散、调用链过长或理解成本升高，回收并合并不必要的结构。
- 当当前改动附近出现高重合逻辑时，能合并就合并，能抽公共能力就抽公共能力。
- 避免当前功能范围内相同能力出现多个实现；如果暂时不能合并，说明原因，并把新实现限制在当前功能边界内。
- 只在需要判断复用、合并或遵循项目模式时，查找相关页面、弹窗、组件、ViewModel、Repository、主题、文案工具或路由方式。
- 不主动引入新的状态管理、依赖注入、路由或网络框架；项目已有约定优先。

## 避免过度设计

- 不为了形式上的分层硬拆方法、类或文件。没有复用价值、也没有明确语义收益的一次性简单逻辑，不要强行包装成独立结构。
- 单段 `try/catch`、一次性数据转换、单处条件分支或几行顺序逻辑，如果放在原位置更直接，就不要仅为了“更规整”再拆一个方法。
- 发现复杂度上升时，优先合并无收益抽象、收回一次性封装、缩短命名和理顺职责，而不是继续加条件分支。

## 页面与弹窗拆分

- 页面级和弹窗级文件通常不超过 400 行。超过约 300 行时，主动判断是否需要拆分。
- 重复 UI 片段拆成小组件。
- 独立业务块拆成私有 Widget 或单独文件。
- 和视图无关的逻辑不得堆在页面类中。
- 除非用户明确说明页面非常简单，可以把全部逻辑写在页面中；否则复杂页面必须拆分。

## 质量边界

实现时持续检查这些问题：

- 一个类是否承担了多个原因变化的职责。
- 一个方法是否同时做请求、转换、校验和 UI 组装。
- 是否为了形式整洁，把一次性简单逻辑、单段 `try/catch` 或一行转换拆成了没有复用价值的方法。
- 一个字段是否缺少清晰生命周期或所有权。
- 文件、类、方法、字段命名是否过长，读完整个名字也没有新增信息。
- 同一页面是否出现多个相似的 `loading`、`empty`、`error`、`submit` 状态实现。
